| `CronHealthAnalyzer` | v3 Layer 1 感知增强 | 已有 |
| `SkillsQualityAudit` | v3 Layer 1 感知增强 | 已有 |
| `GitChangeAnalyzer` | v3 Layer 1 感知增强 | 已有 |
| `ReportGenerator` | v3 Layer 3 报告生成 | 已有 |

**证据**:  
- `v2/README.md` 明确写出「v2 的变化：分级执行、低风险自动、中风险出方案、高风险标记讨论」

**影响**:

2. 重新定义 v3 = 「将 v2 的独立分析器改造为 v1/v2/v3 三层集成架构」
3. 具体资产清单：`v2` 目录中的 `AutoFixer`, `EvolutionVerifier`, `RalphLoop`, `ActionPlanner` 四个类可直接复用或封装

### P0-3: 缺少故障恢复和事务保障设计

**问题描述**:  
方案描述了「每个修复执行后立即验证」，但缺少：

1. **执行中的故障恢复**: 如果一个修复执行到一半崩溃（如 `patch` 打到一半、terminalsession 未关闭），系统状态如何恢复？
2. **幂等性保障**: 同一修复被重复执行，是否会破坏系统（无锁机制的 cron 可能重叠）？
3. **部分成功处理**: 批量修复中部分失败、部分成功，如何保证数据一致性？
4. **回滚机制**: 没有提到任何修复失败后的回滚策略

**证据**:  
- v1 有 `wal_protocol.py` 和 `working_buffer.py`，但 v3 方案未提及是否沿用或增强
- cron 驱动天然缺少事务边界

**影响**:  
不加事务保障的自动修复可能导致系统状态不可逆地损坏。

**建议**:  
1. 在执行层增加「修复事务管理器」：记录每个修复的开始/步骤/完成/回滚
2. 利用已有的 WAL 协议，扩展为修复操作的持久化日志
3. 为每种可自动修复类型定义「回滚操作」或「安全检查前置条件」
4. 添加 `fix_lock.json` 防并发执行

## P1 级问题（重要）

### P1-1: Phase 1 范围过宽，缺少明确的完成标准和验收条件

**问题描述**:  
方案中 Phase 1 定义为「进化决策引擎（执行引擎 + 验证引擎 + 改造 cron prompt）」。但这至少是 4 个独立交付物：

| 子任务 | 复杂度评估 | 依赖 |
|--------|-----------|------|
| 执行引擎 (AutoFix v2) | 中 | 需先完成问题分类 + 安全护栏 |
| 验证引擎 (Verifier v2.1) | 中 | 需先定义验证指标 |
| 改造 cron prompt | 低 | 独立 |
| 问题分类与风险评估 | 低-中 | 依赖感知层输出解析 |

无完成标准 → 无法判断「Phase 1 是否完成」。

**影响**:  
迭代规划模糊，容易「做着做着发现还要做 X」导致 Phase 1 无限膨胀。

**建议**:  
1. 将 Phase 1 拆分为：
   - **Phase 1a**: 改造 cron prompt + 集成 v2 现有执行/验证到主流程
   - **Phase 1b**: 增强安全护栏 + 白名单 + 熔断
   - **Phase 1c**: 扩展修复类型（磁盘清理、配置修复）
2. 为每个 Phase 子项定义硬性完成标准（如：至少 3 种问题类型能通过执行→验证环路）

---

### P1-2: 安全护栏描述过于粗略，缺少「熔断」的具体触发条件

**问题描述**:  
方案提到「安全护栏：白名单修复类型、熔断机制、用户确认」，但：
- 熔断的触发阈值是什么？（连续失败 N 次？同一文件修改超过 M 次？）
- 熔断后如何恢复？（手动？自动冷却后重试？）
- 用户确认的超时处理？（cron 运行在凌晨，无人值守时如何处理需要确认的问题？）
- 白名单是全局还是按问题类型的？

**建议**:  
为熔断机制补充以下具体设计：
1. **熔断触发条件**:  
   - 同一修复类型连续失败 ≥ 3 次 → 该类型熔断 24h  
   - 同一文件在 1 小时内被修改 ≥ 2 次 → 文件级熔断  
   - 整体失败率 ≥ 50%（最近 10 次修复） → 系统级熔断  
2. **恢复机制**:  
   - 时间冷却：24h 后自动半开（允许 1 次尝试）  
   - 手动重置：通过状态文件强制恢复  
3. **无人值守确认**:  
   - 所有需用户确认的问题，默认「跳过」，记录到报告  
   - 仅在确保安全的范围内执行自动操作

### P1-3: 方案未评估 LLM 决策的可靠性风险

**问题描述**:  
Layer 2 决策层依赖 LLM 进行「问题分类」和「方案生成」，但：
- LLM 对系统状态的判断可能出错（幻觉、过度自信）
- 不同 LLM 版本/模型可能导致不一致的决策
- 没有回退策略：如果 LLM API 不可用，决策层是否降级为仅报告？

**证据**:  
当前 v1/v2 都是基于确定性的规则（正则、结构化数据解析），而 v3 引入了 LLM 决策层，这是架构上的本质变化。

**建议**:  
1. 决策层采用「规则优先，LLM 补充」的混合策略：
   - 确定性问题（磁盘清理阈值、技能文档缺失）→ 规则直接分类
   - 模糊问题（工作流优化、趋势预测）→ LLM 辅助
2. LLM 决策加入置信度评分，低置信度自动降级为「仅报告」
3. 增加 LLM API 不可用的降级路径：退化为当前 v2 的确定性决策能力

---

### P1-4: 「学习层」在 Phase 2 才实现，但 Phase 1 就需要反馈闭环

**问题描述**:  
方案将「学习层（修复历史追踪、干预效果评估、策略优化）」放在 Phase 2，但：
- Ralph Loop 本身就依赖历史迭代数据来判断是否继续
- 熔断机制需要修复成功率历史
- 「知识积累」是自动修复质量提升的核心驱动

没有学习层 → Phase 1 的 Ralph Loop 每次都是从零开始 → 与 v2 现有的 Ralph Loop 能力持平，没有增量价值。

**建议**:  
1. 将「修复历史追踪」的最小版本提到 Phase 1：
   - 只需要一个 JSON 文件记录每次修复的类型、结果、耗时
   - 不需要完整的策略优化
2. Phase 2 再扩展为效果评估和策略优化

## P2 级问题（一般）

### P2-1: 缺少 Hermes 工具在 cron 环境中的可行性验证

**问题描述**:  
方案说「修复使用 Hermes 已有工具（patch, write_file, terminal, skill_manage）」，但：
- cron 任务运行在非交互式 shell 中，Hermes Agent 的工具是否可在纯 Python 环境中调用？
- `patch` 和 `skill_manage` 是 Hermes Agent 的 MCP 工具，还是独立的 CLI？
- 当前 `evolve_analysis.py` 是纯 Python 脚本，由 crontab 直接调用 Python，没有 Hermes Agent 的 tool 环境

**证据**:  
查看 `evolve.sh` 和 `v2/hermes_self_analyzer.py`，自动修复使用的是 `SKILL.md` 模板写入（标准文件 I/O），没有使用 Hermes Agent 的 MCP 工具。

**建议**:  
1. 明确 Hermes 工具的调用方式：
   - 如果工具需要通过 MCP 协议 → 需要改造 cron 任务为 Hermes Agent 子进程或 API 客户端
   - 如果工具是独立 CLI → 需要验证是否在 cron 环境中有正确的 PATH 和环境变量
2. 作为备选，保留纯 Python 的修复实现（如 v2 的模板写入）

---

### P2-2: 可自动修复的问题范围缺少对「依赖关系」的考虑

**问题描述**:  
修复类型以独立维度列出，但实际可能存在依赖：
- 修复技能文档 → 可能触发技能加载 → 可能影响技能管理工具
- 磁盘清理 → 可能删除了正在运行的进程使用的临时文件
- 配置修复 → 可能要求重启服务，但重启服务被标记为「仅报告」

**影响**:  
独立考虑修复类型可能导致修复 A 破坏了 B 正常运行的环境。

**建议**:  
1. 为每种修复类型添加「前置检查」和「后置影响」
2. 修复执行排序：先安全无关（文档补全）→ 再影响系统状态（磁盘清理）→ 最后配置变更
3. 配置修复和磁盘清理等中度风险操作执行前先做快照备份

---

### P2-3: 缺少监控和告警指标

**问题描述**:  
Agent Evolution 本身也是一个需要被监控的系统：
- 自动修复本身的成功率是多少？
- 修复引入的新问题率和频率？
- 系统是否在「退化」（修复 A 导致问题 B，修复 B 导致问题 C…）？

方案没有定义以下监控指标：
- 修复成功率（按类型）
- 修复引入的回归问题率
- 系统健康度趋势（多周期对比已有实现但未在 v3 方案中体现）
- Agent Evolution 自身执行成功/失败

**建议**:  
至少增加以下指标采集：
1. `agent_evolution_auto_fix_total` 按类型 counter
2. `agent_evolution_auto_fix_success_rate` 按类型
3. `agent_evolution_circuit_breaker_triggered` 熔断触发次数
4. 每次运行输出自身健康状况到日志

## P3 级问题（建议）

### P3-1: 缺少「测试模式」或「试运行」机制

目前设计是 cron 驱动直接作用于生产环境。建议增加：
- 试运行模式：执行但不实际 apply，仅记录「会做什么」
- 可在 `--dry-run` 参数下看到完整的修复计划和预期效果

### P3-2: 报告格式升级未考虑

v3 执行后生成的报告是否沿用现有 Markdown 格式？建议增加「执行摘要」section，突出显示：
- 尝试了多少修复，成功多少，失败多少
- 被熔断阻止了哪些操作
- 待用户确认的事项

### P3-3: 技能库分析的双路径冲突

v1 (`evolve_analysis.py`) 和 v2 (`hermes_self_analyzer.py`) 各有独立的技能库扫描逻辑，v3 需要统一为一条路径，避免重复扫描和结果不一致。

## 实施建议汇总

### 优先级调整

1. **立即做**（P0 修正）:
   - 建立 v2 → v3 继承基线，明确哪些代码直接复用
   - 替换 `hermes-scan.py` 为实际文件名
   - 设计修复事务管理和回滚机制

2. **Phase 1 核心**:
   - 将 v2 的 `AutoFixer` + `EvolutionVerifier` + `RalphLoop` 集成到主流程
   - 熔断机制具体化（触发条件、恢复策略）
   - 决策层混合策略（规则优先 + LLM 辅助 + 降级）
   - 最小化的修复历史追踪（JSON 日志）

3. **Phase 1 可选**:
   - Hermes 工具调用接口封装
   - 依赖检查排序
