## P0 — 阻断性问题

### P0-1: 「只报不改」的问题在 v3 方案中被部分继承 — 决策层到执行层之间存在逻辑裂痕

**位置**: 决策层 → 执行层 接口  
**严重性**: P0

**问题描述**:  
v3 方案提出三层架构，但方案描述中只列出「可自动修复的问题类型」列表，没有定义**决策层输出的数据结构**和**执行层输入的契约接口**。现有 v2 代码使用 `ActionPlanner` 输出的 `{id, type, priority, execution, steps, auto_fix}` 结构，其中 `execution` 字段标记 `auto/review/discuss` 三级。v3 方案如果复用 v2 的分类逻辑，则:

1. 方案未定义「问题类型 → 风险等级 → execution 模式」的映射表（如 `skill_documentation → low → auto`），导致决策逻辑硬编码
2. 没有标准化输出格式，执行层需要做类型推断，容易出现分类漂移
3. 现有 v2 代码中 `_plan_skill_improvements()` 对缺失文档标记为 `auto`，但对空文档标记为 `review` — 这种粒度在 v3 方案中没有体现

**风险**: 新增可修复类型时，必须在决策层和执行层同时修改代码，违反开闭原则。

**建议**:  
1. 明确定义决策层输出的统一 Schema (JSON Schema 或 Protocol Buffers)，包含 `problem_type`, `risk_level`, `execution_mode`, `auto_fix_action` 等字段
2. 建立一个「问题类型 → 风险等级 → execution 模式」的配置式映射表（YAML/JSON），而非硬编码
3. 在方案中补充 `DecisionLayerOutput` 的数据结构定义

### P0-2: 熔断机制缺乏状态持久化和跨周期记忆

**位置**: 执行层 — 熔断机制  
**严重性**: P0

**问题描述**:  
方案中提到的「熔断机制」只在概念层面描述，没有具体实现设计。关键缺陷：

1. **无持久化计数器**: 熔断需要记录「单位时间内修复失败次数」，但方案没有说明这个状态存在哪里。每次 cron 触发是一个全新进程，内存计数器在进程结束后丢失
2. **无半开状态 (Half-Open)**: 标准熔断模式（Closed → Open → Half-Open → Closed）在方案中完全缺失。一旦熔断触发，没有任何机制说明「何时解除熔断」
3. **无跨周期记忆**: 假设某个修复连续失败 3 次触发熔断，但下次 cron 进程启动时熔断状态丢失，又会重新尝试 — 这正是 v2 中已有 `EvolutionState` 机制但未升级到 v3 的问题

**风险**: 熔断机制形同虚设，修复-失败-重试的死循环无法阻断。

**建议**:  
1. 熔断状态持久化到文件系统文件（如 `~/.hermes/evolution/circuit-breaker.json`），包含 `problem_type`, `failure_count`, `state`, `last_failure`, `cooldown_until`
2. 实现三态模型:
   - **Closed**: 正常执行修复
   - **Open**: 连续 N 次失败后触发，M 分钟内跳过该类型修复
   - **Half-Open**: 冷却期后允许单次尝试，成功则回到 Closed，失败则回到 Open
3. 在 `~/.hermes/cron/` 中已有 `ralph-state/` 和 `ralph-config.json` 的先例，熔断状态可复用该机制

### P0-3: 验证机制存在「假阳性」系统性风险 — 缺失差异归因逻辑

**位置**: 执行层 — 验证引擎  
**严重性**: P0

**问题描述**:  
方案提出「每个修复执行后立即验证」，但 v2 的 `EvolutionVerifier.verify()` 实现存在根本性缺陷：

```python
# v2 现有代码（需要升级到 v3）
def verify(self, before_state, auto_fix_results):
    cron_health = CronHealthAnalyzer().analyze()  # 重新分析
    skills_audit = SkillsQualityAudit().analyze()  # 重新分析
    # 然后比较 before 和 after 的指标
```

这里的问题是:
1. **非原子性验证**: `before` 和 `after` 是两个独立分析周期的快照，中间可能有其他系统变化（如其他 cron 任务、用户操作），`after` 指标的改善/恶化不能归因于本次修复
2. **指标噪声**: cron 任务的健康状态有自然波动，v2 使用 `success_rate` 的 70% 阈值阈值判断正常/降级，但这个阈值没有考虑基线波动
3. **验证样本量不足**: 验证只做一次重新分析，如果是偶发性问题，单点验证不可靠

**风险**: 修复实际上未生效但验证报告「通过」(假阳性)，或修复实际生效但验证报告「失败」(假阴性，由其他任务干扰导致)。

**建议**:  
1. 引入「差异归因」机制: 在验证结果中标记不确定性来源，如 `{"verified": True, "confidence": 0.7, "confounding_factors": ["other_cron_activity"]}`
2. 对于偶发性问题，建议多次验证（如连续 2 次 re-analysis 都通过才标记为有效）
3. 在方案中补充「验证置信度」概念: 低置信度的验证结果应标记为 `partial` 而非 `passed`
4. 考虑在一次 cron 周期内完成「修复 → 等待 → 验证」而非跨周期比较

## P1 — 重要缺陷

### P1-1: 「disk cleanup」中风险修复的提交流程缺失具体设计

**位置**: 决策层 — 需批准的中风险修复流程  
**严重性**: P1

**问题描述**:  
方案提到「中风险需批准」的修复需要用户确认，但没有定义:
1. **交互通道**: 通过什么渠道通知用户？(飞书消息？邮件？终端？)
2. **超时策略**: 用户如果 24 小时内不回复，是默认拒绝、默认执行、还是保持等待？
3. **回滚准备**: 如果用户批准后执行失败，v3 方案只提了验证，没有回滚预案
4. **批量确认**: 多个中风险修复同时触发时，是逐个确认还是汇总展示？

**风险**: 用户交互流程不清晰导致修复卡死。现有 v2 方案中的 `review` 模式实际上只是生成报告，并没有真正阻塞等待用户回复。如果 v3 语义变为「等待用户确认」，则需要重新设计整个异步交互流程。

**建议**:  
1. 明确定义用户交互流程的状态机:
   ```
   [发现中风险问题] → [生成修复方案] → [发送飞书消息/邮件等待确认]
       → [用户批准] → [执行] → [验证] → [通知结果]
       → [用户拒绝] → [记录日志，跳过]
       → [超时 N 小时] → [默认拒绝，记录待处理]
   ```
2. 实现一个「待批准队列」机制，状态持久化到文件
3. 方案应明确标注是否需要新增常驻消息监听（违背「不新增常驻进程」的设计决策），或改用 cron 轮询待批准队列

### P1-2: 学习层仅停留在概念描述，缺乏具体机制设计

**位置**: 学习层  
**严重性**: P1

**问题描述**:  
方案中的「学习层」只列了三个功能: 修复历史追踪、干预效果评估、策略优化。但没有:
1. **学习数据从哪里来？** 是每次修复后自动生成训练样本，还是手动标记？
2. **策略优化如何实现？** 策略优化需要对比不同修复方案的 ROI（成功率、修复时间、副作用），这需要跨周期的纵向数据
3. **学习层和三层之间如何交互？** 是学习层被动接收数据（写日志），还是主动影响决策层的分类逻辑（模型更新）？
4. **学习存储格式？** 现有 v2 的 `knowledge-base.json` 只有最佳实践/失败教训/用户偏好三类，没有修复策略的量化评估

**风险**: 学习层沦为空洞概念，无法形成「修复 → 评估 → 优化」的正向飞轮。

**建议**:  
1. 将学习层的数据模型具体化，定义 `RepairRecord` 结构:  
   ```json
   {
     "problem_type": "disk_cleanup",
     "repair_strategy": "remove_cron_old_output",
     "risk_level": "medium",
     "execution_result": "success/failure/partial",
     "verification_confidence": 0.8,
     "execution_duration_ms": 2300,
     "side_effects": ["cron_output_size_reduced"],
     "timestamp": "..."
   }
   ```
2. 策略优化可先简化为「根据修复成功率调整 risk_level」的规则引擎，再逐步演进到机器学习模型
3. 在架构图中，学习层应有反馈箭头回到决策层，形成闭环

### P1-3: 「process anomaly」和「service down」高风险类型只报告，但缺少分级告警设计

**位置**: 决策层 — 问题分类  
**严重性**: P1

**问题描述**:  
方案将「进程异常」和「服务宕机」标记为高风险→仅报告。但:
1. **仅报告 ≠ 什么都不做**: 报告本身需要渠道和格式。是生成一条日志、一个飞书消息，还是创建一个 GitHub Issue？
2. **无告警收敛**: 如果服务连续 7 天宕机，系统每天生成 1 份报告，没有「已告警过、不需要重复报告」的收敛逻辑 — 这导致告警疲劳
3. **无升级机制**: 如果某问题连续 N 天出现，应该升级告警级别（从飞书消息 → 电话/短信）

**风险**: 高风险问题被「报告」淹没，运维人员对重复告警视而不见。

**建议**:  
1. 定义报告渠道矩阵: `高风险(P0) → 飞书消息 + 报告`, `中风险(P1) → 报告 only`
2. 实现告警收敛: 已报告过的高风险问题，如果 7 天内状态无变化，不再重复告警（但在报告中保留记录）
3. 实现升级机制: 连续 N 天未解决的 P0 问题，生成「告警升级记录」

### P1-4: 安全护栏中的「白名单修复类型」缺少动态更新机制

**位置**: 决策层 — 安全护栏  
**严重性**: P1

**问题描述**:  
方案提到「白名单修复类型」，但没有说明:
1. **白名单存储格式** 和 **位置**: 是硬编码在代码里、存 JSON 文件、还是 Hermes skill 配置？
2. **如何更新白名单？** 方案中说「用户可以通过调整配置文件来启用/禁用特定修复类型」，但用户交互接口没有定义
3. **白名单和 v3 新增修复类型的关系**: 新增一种修复类型时，需要同时更新白名单 — 这个更新流程是什么？

**风险**: 白名单机制执行在数据层（硬编码）而非配置层，每次修改需要改代码。

**建议**:  
1. 白明单使用独立 YAML/JSON 配置文件（如 `~/.hermes/evolution/repair-whitelist.yaml`），包含:
   ```yaml
   allowed_repairs:
     skill_documentation:
       enabled: true
       max_auto_per_run: 5
     disk_cleanup:
       enabled: true
       requires_approval: true
     cron_config:
       enabled: false  # 默认禁用高风险类型
   ```
2. 为白名单提供 CLI 管理命令（或 Hermes skill `evolution_manage`）

### P1-5: 方案未定义「修复副作用」的检测和处理机制

**位置**: 执行层 — 自动修复引擎  
**严重性**: P1

**问题描述**:  
执行修复可能带来副作用:
- `disk_cleanup`: 删除了还在使用的缓存，导致其他任务变慢
- `cron_config`: 修改 `enabled_toolsets` 导致某个任务功能缺失
- `patch` 修复配置: 修改了配置文件的格式，导致解析器无法加载

方案只提了「事后验证」，但没有定义:
1. **如何检测副作用？** 修复前后除了目标指标，还需要监控哪些关联指标？
2. **副作用回滚**: 如果检测到副作用，是否有回滚预案？
3. **副作用等级**: 轻微副作用（如性能下降 5%）vs 严重副作用（如系统不可用）的处理策略不同

**风险**: 一个修复解决一个问题的同时，可能引入一个或多个新问题，而方案无检测手段。

**建议**:  
1. 为可自动修复类型定义「关联监控指标清单」:  
   - `disk_cleanup` → 监控 cron 输出目录大小、磁盘使用率增幅
   - `cron_config` → 监控 cron 任务成功率、任务执行时间
   - `patch` → 监控配置文件语法校验、服务重启状态
2. 实现「验证前快照 → 修复 → 验证结果 + 对比关联指标 → 检测到副作用 → 回滚」的完整流程
3. 回滚可以是自动的（低风险修复）或提交通知用户（中风险）

---

### P1-6: 「Cron 驱动」设计下，修复合集 (batch) 的原子性问题

**位置**: 执行层 — 整体执行模型  
**严重性**: P1

**问题描述**:  
方案说所有修复都是 cron 驱动的（不新增常驻进程）。但一个 cron 周期内，系统可能发现并执行多个修复。问题:
1. **原子性**: 如果 5 个修复中的第 3 个失败了，前 2 个已经生效，后 2 个还没执行 — 系统应该回滚前 2 个、继续执行后 2 个、还是标记为「部分完成」？
2. **依赖关系**: 修复之间可能有隐式依赖（如先修复配置错误再重启服务），方案没有考虑依赖拓扑
3. **部分完成的一致性**: 执行到一半被 cron 超时打断，系统的状态如何？下次 cron 周期如何识别「未完成的事务」？

**风险**: 多个修复交叉执行导致系统进入不一致状态。

**建议**:  
1. 引入「事务性执行」概念: 高风险修复单次只执行一个；低风险修复可按批次执行
2. 实现依赖拓扑排序（使用 DAG），确保修复执行顺序正确
3. 持久化执行状态: 每次 cron 周期开始前读取状态文件，判断是否有未完成的修复合集

## P2 — 一般问题

### P2-1: `hermes-scan.py` 作为感知层的数据输出格式未定义

**位置**: 感知层  
**严重性**: P2

**问题描述**:  
方案说「Layer 1: 感知层 — 现有 hermes-scan.py 不变」。但 `hermes-scan.py` 的实际输出格式是什么？如果它的输出是一个 Markdown 报告，决策层需要解析 Markdown 结构化数据 — 这非常脆弱，回到 v1 的正则匹配老路。

当前 v2 代码已经有 `CronHealthAnalyzer`、`SkillsQualityAudit`、`GitChangeAnalyzer` 三个独立分析器，每个都输出结构化的 Python dict。如果 v3 感知层复用 `hermes-scan.py` 的 Markdown 输出，反而是倒退。

**建议**:  
1. 不要复用 `hermes-scan.py` 的原始输出格式，而是输出标准 JSON Schema（或直接复用 v2 的 dict 结构）
2. 如果必须保留 `hermes-scan.py`，在其之上加一层结构化解析器，将 Markdown 转为结构化数据
3. 定义感知层输出的 Schema，包含 `cron_health`, `skills_audit`, `disk_usage`, `process_status`, `config_status` 等字段

---

### P2-2: 风险评分的计算逻辑未定义

**位置**: 决策层 — 风险评估  
**严重性**: P2

**问题描述**:  
方案提到「风险评估」，但没有定义评分模型。当前 v2 的风险判断是硬编码的（如 `无文档 → auto`, `cron 失败 → review`, `服务宕机 → discuss`）。v3 方案如果希望引入动态风险评估，需要解决:
1. **评分因子**: 影响范围、发生频率、恢复难度、数据损失风险
2. **评分公式**: 这些因子如何加权组合得到最终风险等级？
3. **动态调整**: 学习层如何反馈修正评分权重？

**建议**:  
1. 明确 v3 初版是否保持 v2 的静态分类（类型→风险映射表），或者引入评分模型
2. 如果保持静态映射，方案应直接列出完整的「问题类型 → 风险等级」映射表（而不是示例说明）
3. 如果引入评分模型，需要补充评分因子的定义和计算方式

---

### P2-3: 方案没有定义修复执行超时和重试策略

**位置**: 执行层  
**严重性**: P2

**问题描述**:  
每个修复执行可能耗时不同，cron 任务有总执行时间限制。方案没有定义:
1. 单个修复的超时时间
2. 修复失败的重试策略（指数退避？重试次数？）
3. 整体 cron 任务的超时保护（防止一个修复耗完所有时间）

**建议**:  
1. 定义每个 repair type 的预计执行时间、超时时间
2. 参考 v2 的 `RalphLoop` 实现重试（最多 3 次，提前通过则停止）
3. cron 入口加总超时看门狗 `signal.alarm()` 或异步超时

---

### P2-4: 方案没有考虑「并发安全」问题

**位置**: 全局  
**严重性**: P2

**问题描述**:  
所有修复由 cron 驱动，但:
1. 如果 cron 任务执行时间过长（如 30 分钟），和下一次 cron 触发（5 分钟一次）重叠怎么办？
2. 多个 cron 任务同时修改同一个文件（如 `enabled_toolsets`），导致竞争条件
3. `hermes-scan.py` 执行期间，修复引擎同时修改系统状态，数据采集和修复操作之间的一致性

**建议**:  
1. 引入文件锁机制（PID file 或 flock），防止并发执行
2. cron 执行频率应配置化，确保最坏情况下也有 buffer
3. 声明「cron 任务执行期间串行化」的约束

## P3 — 低优先级改进建议

### P3-1: 方案缺少人工干预的 API/接口设计

**位置**: 全局  
**严重性**: P3

**建议**: 如果用户想手动触发某个修复、查看修复历史、或修改白名单配置，当前方案没有提供入口。建议:
- 提供 CLI 包装脚本（如 `evolve.sh --repair disk_cleanup`）
- 或通过 Hermes skill 暴露交互接口（`evolution_manage` skill）

### P3-2: 报告生成应区分「人类可读」和「机器可消费」两种格式

**位置**: 执行层 — 报告生成  
**严重性**: P3

**建议**: 当前 v2 只生成 Markdown 报告（人类可读）。v3 应增加结构化的 JSON 报告输出，供学习层消费。Markdown 报告应该成为「人类可读摘要」而非唯一输出。

### P3-3: 方案提到的「Ralph Loop」在 v3 和 v2 中的定位需要区分

**位置**: 执行层  
**严重性**: P3

**建议**: v2 已经有一个完整的 `RalphLoop` 类做迭代进化控制（分析→改进→验证→循环）。v3 方案提到「执行层包含 Ralph Loop」，但没说明 v3 的 Ralph Loop 和 v2 有何不同。建议在方案中明确:
- v2 Ralph Loop: 在单个 cron 周期内进行多轮迭代
- v3 Ralph Loop: 跨 cron 周期的学习进化，或复用 v2 的机制

## P4 — 信息性观察

### P4-1: 方案整体与 v2 代码的演进关系不明确

**观察**: v3 方案的「三层架构 + 学习层」和 v2 的模块结构（`CronHealthAnalyzer`, `SkillsQualityAudit`, `ActionPlanner`, `AutoFixer`, `EvolutionVerifier`, `RalphLoop`）有大量重合。建议在方案中增加一个「v2 → v3 映射表」，明确哪些模块保留、哪些重构、哪些新增。否则开发时容易重复造轮子或冲突。

### P4-2: 建议补充一个「修复生命周期状态图」

**观察**: 从问题发现到修复生效，经过感知→决策→（用户确认）→执行→验证→（学习）多个阶段。每个阶段有多种状态（pending/processing/success/failure/rollback）。建议补充一个完整的修复生命周期状态图，作为方案的可视化补充。

## 总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| **功能完整性** | ⚠️ 6/10 | 核心修复链路由清晰定义，但边界条件、异常处理、副作用检测缺失严重 |
| **逻辑正确性** | ⚠️ 5/10 | 安全护栏和熔断机制的基本思路正确，但缺乏具体实现设计和容错策略 |
| **安全护栏** | 👌 7/10 | 白名单、用户确认的思路正确，但白名单更新机制、熔断持久化缺失 |
| **异常处理** | ❌ 3/10 | 超时、并发、部分失败、回滚等场景几乎没有覆盖 |
| **用户交互** | ❌ 4/10 | 中风险需批准的交互流程不完整，缺超时策略和交互通道定义 |
| **与 v2 代码的关系** | ⚠️ 5/10 | 看似重设计，实则和 v2 已有模块高度重合，缺少演进路径映射 |

**关键风险**: v3 方案「看起来合理」但实际缺失了大量实现细节。最核心的 3 个 P0 问题（决策→执行接口契约、熔断持久化、验证假阳性）如果不解决，v3 实现后的行为与「只报不改」的现状改进有限。

建议在 Phase 1 至少解决 P0-1 和 P0-3，再进入实现阶段。
